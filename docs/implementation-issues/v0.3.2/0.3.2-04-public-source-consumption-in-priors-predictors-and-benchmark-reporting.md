# Issue 0.3.2-04

## Status

Complete on the current branch.

## Title

Public-source consumption in priors, predictors, and benchmark reporting

## Objective

Wire aggregate public stats and normalized public-run summaries into repo-native priors, predictor reports, and benchmark outputs with explicit provenance, freshness, and sample-size diagnostics.

## Problem

After source import and normalization land, the repo still needs a disciplined consumption layer. External public data should influence priors and evaluation only where it is structurally appropriate, and every downstream summary should say which public source, snapshot window, and sample size it used. Without those rules, external signals become hard to trust and easy to overfit.

## In Scope

- public-source prior inputs for reward, shop, route, or other strategic scoring hooks where evidence is strong
- predictor / benchmark summaries that compare repo behavior against public aggregate and public-run-derived baselines
- freshness, provenance, and sample-size reporting in summaries and markdown outputs
- config surfaces and tests

## Out Of Scope

- using public runs as the primary offline RL or BC dataset
- live web calls during evaluation or training
- generic public-data fusion across unsupported sources

## Dependencies

- `0.3.2-01` aggregate snapshot lineage
- `0.3.2-03` normalized public-run strategic summaries
- existing community prior, predictor reporting, and benchmark suite infrastructure

## Affected Areas

- `src/sts2_rl/collect/`
- `src/sts2_rl/predict/`
- `src/sts2_rl/train/`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## Acceptance Criteria

- policy and reporting configs can opt into public aggregate or public-run-derived signals through explicit local artifact paths
- benchmark and predictor reports surface provenance, freshness, coverage, and sample-size diagnostics for every public-data-backed metric
- route / reward / shop priors can consume derived public strategic stats without depending on live source fetches
- alignment reports distinguish aggregate-card baselines from run-derived strategic baselines
- tests cover config wiring, artifact loading, and summary rendering

## Implementation Breakdown

### 1. Prior consumption

- extend prior loaders to accept normalized public strategic summaries where appropriate
- keep aggregate card priors and public-run priors distinct in trace output

### 2. Predictor and benchmark reporting

- add public-source comparison sections to benchmark and predictor markdown / JSON summaries
- expose source snapshot windows, sample sizes, and freshness

### 3. Config and CLI surfaces

- add explicit artifact-path config knobs instead of hidden auto-discovery
- keep external data optional and local-first

### 4. Guardrails

- fail clearly when required artifacts are stale, missing, or too small
- avoid silent fallback from public-run-derived metrics to aggregate-card metrics

### 5. Tests and docs

- cover metric provenance, stale-source behavior, and reporting layout
- document when public-source signals should and should not influence decisions

## Deliverables

- prior, predictor, and benchmark integrations for public-source artifacts
- provenance-rich reporting outputs
- tests and roadmap / issue documentation

## Delivered On The Current Branch

- community prior loaders now accept either `community_card_stats` aggregate artifacts or `public_run_strategic_stats` card artifacts through explicit local paths
- route priors now consume `strategic-route-stats.jsonl` through explicit `route_source_path` wiring and influence both legacy map scoring and route-prefix planning
- prior traces now distinguish aggregate-card vs run-derived strategic sources through `artifact_family`, `source_name`, `snapshot_date`, and sample-size metadata
- predictor calibration, ranking, and benchmark-compare reports now accept explicit public aggregate / public-run artifact paths and emit provenance, freshness, and sample-size diagnostics in `summary.json` and markdown
- benchmark suite case and suite summaries now surface configured public-source diagnostics and aggregate them across eval / compare cases
- runtime job manifests, benchmark manifests, and collect / train / eval CLI commands now expose explicit route-prior and freshness guard knobs
- tests cover strategic card/route artifact loading, route-prior decision impact, predictor report diagnostics, benchmark summary wiring, and manifest path resolution
