# Issue 0.3.2-02

## Status

Completed on 2026-04-13.

## Title

STS2Runs public run archive contract and incremental sync

## Objective

Create a dedicated `public_run_archive` artifact family for `STS2Runs` so the repo can ingest public run listings and run-detail payloads incrementally, resumably, and with stable dedupe semantics.

## Problem

Public run archives are structurally different from aggregate card snapshots. They carry run-level metadata, act history, deck / relic summaries, map path, and encounter detail. Reusing `community_card_stats` for this data would collapse two different source types into one schema and make later normalization brittle. The repo also lacks incremental sync rules for page cursors, run-detail backfill, and duplicate protection.

## In Scope

- raw archive contract for paged run listings and per-run detail payloads
- manifest files for source lineage, cursor progress, fetch windows, and backfill status
- stable dedupe keys using source run ids plus content hashes or fallback identity fields
- resumable sync flow that can continue from prior manifests
- CLI commands and regression tests

## Out Of Scope

- derived strategic summaries from public runs
- predictor or prior consumption
- treating public runs as step-level trajectory supervision

## Dependencies

- current artifact conventions, summary generation, and CLI patterns
- source validation from `STS2Runs` list and detail endpoints

## Affected Areas

- `src/sts2_rl/data/`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## Acceptance Criteria

- the repo can sync `STS2Runs` page-by-page into a local archive without refetching all historic data on every run
- archive manifests preserve source URLs, page windows, fetch timestamps, cursor positions, and detail-fetch coverage
- duplicate runs are prevented by stable dedupe rules and surfaced in summary diagnostics
- detail payloads can be backfilled after list-page discovery without corrupting earlier archive sessions
- sync can resume from prior manifests after interruption
- tests cover pagination, resume behavior, dedupe, and detail backfill

## Implementation Breakdown

### 1. Raw archive contract

- define filenames and typed records for list pages, run-detail payloads, and sync manifests
- keep raw payloads intact enough for later re-normalization

### 2. Incremental sync engine

- support page-based polling with limit / page parameters
- persist cursors and fetch status so syncs can resume safely

### 3. Dedupe and identity

- key archive entries by source run id where available
- retain content hash and fallback identity fields for integrity checks

### 4. Detail backfill

- fetch per-run details after discovery
- track which run ids still need detail payloads and which failed transiently

### 5. CLI, summaries, and docs

- add archive sync and summary commands
- document expected refresh cadence and archive layout

## Deliverables

- `public_run_archive` typed contract and storage layout
- resumable `STS2Runs` incremental sync tooling
- archive summary artifacts with freshness and dedupe diagnostics
- tests and roadmap / issue documentation
