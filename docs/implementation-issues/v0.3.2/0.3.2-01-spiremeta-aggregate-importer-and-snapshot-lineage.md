# Issue 0.3.2-01

## Status

Completed on 2026-04-13.

## Title

SpireMeta aggregate importer and snapshot lineage

## Objective

Add a repo-native importer for `SpireMeta` aggregate card statistics so external aggregate signals enter the existing `community_card_stats` artifact family with explicit source lineage and point-in-time reproducibility.

## Problem

`v0.3.1` created a clean local contract for community card stats, but the repo still depends on manually prepared local files. That is enough for ad hoc imports, but not enough for repeatable refreshes from a known public source. Without a source-aware importer, repeated pulls lose provenance, request parameters, and raw payload context.

## In Scope

- direct import from `SpireMeta` aggregate card endpoints
- source-specific request parameter capture such as character, page, and page size
- snapshot lineage including fetch time, source URL, and raw payload retention
- canonical conversion into existing `community_card_stats` records
- CLI commands and summaries for source-aware aggregate import
- regression tests and documentation

## Out Of Scope

- public run archive sync
- consumption of public run archives
- online polling inside the training loop

## Dependencies

- existing `community_card_stats` models and CLI commands in `src/sts2_rl/data/community.py` and `src/sts2_rl/cli.py`
- current local-first artifact conventions and summary generation patterns

## Affected Areas

- `src/sts2_rl/data/`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## Acceptance Criteria

- the repo can pull aggregate card stats from `SpireMeta` for one or more characters without a hand-written preprocessing script
- imported artifacts preserve source name, source URL, fetch timestamp, snapshot date, request parameters, and raw payload lineage
- canonical outputs remain compatible with `community_card_stats` loaders and current community prior consumers
- summary artifacts expose source coverage, character coverage, page coverage, and freshness metadata
- repeated imports create point-in-time snapshots instead of mutating a single latest-only dataset
- tests cover paging, raw lineage retention, and stable canonical conversion

## Implementation Breakdown

### 1. Source-aware importer

- add a `SpireMeta` importer that pulls aggregate card stats by character and page
- normalize raw API fields into canonical card-stat records

### 2. Snapshot lineage

- persist fetch manifest data including source endpoints, query parameters, fetch time, and raw payload locations
- keep the existing canonical JSONL / CSV / summary outputs

### 3. Incremental refresh contract

- define how repeated pulls create new dated snapshots
- avoid rewriting old snapshots in place
- make source parameters part of the lineage metadata so imports stay comparable

### 4. CLI and docs

- add source-aware import and refresh commands or subcommands
- document when to use direct-source import versus local manual import

### 5. Tests

- cover paged import, metadata retention, and compatibility with existing summary loaders

## Deliverables

- source-aware `SpireMeta` aggregate importer
- lineage-rich `community_card_stats` snapshots
- CLI coverage for aggregate-source import
- tests and roadmap / issue documentation
