# Issue 0.2.5-01

## Title

Runtime strategic-state contract for boss and map planning

## Objective

Expose enough live runtime state to support boss-conditioned route planning and longer-horizon non-combat decisions instead of single-node heuristics.

## Problem

The current runtime contract only exposes the immediate map choices. That is enough for "pick one of the current nodes" logic, but not enough for strategic planning. The controller cannot condition on the current act boss, cannot score the distance to rest/shop/elite before that boss, and cannot compare whole paths because the map graph is not available.

## In Scope

- expose act-level strategic state such as boss identity and act number
- expose a full map graph or an equivalent route-planning view, not just `available_nodes`
- expose current map position, reachable edges, and node states needed for short-horizon route search
- persist the new fields through the typed env models and trajectory schema
- keep backward-compatible fallbacks where practical, but make the richer contract first-class

## Out Of Scope

- training a new strategic model in this issue
- replacing the transport away from `STS2-Agent`
- full savestate simulation or headless execution

## Dependencies

- existing typed env models and trajectory schema
- the local `STS2-Agent` fork or upstream integration point where state endpoints are defined

## Affected Areas

- `STS2-Agent` runtime state endpoint
- `src/sts2_rl/env/models.py`
- `src/sts2_rl/env/client.py`
- `src/sts2_rl/data/trajectory.py`
- `tests/`

## Acceptance Criteria

- live state includes boss identity or an explicit absence marker
- live state includes a route-plannable map structure beyond the current frontier
- typed models and dataset summaries can ingest the new fields without ad hoc JSON parsing
- trajectory artifacts preserve the strategic fields for later replay and offline analysis
- tests cover parsing, schema evolution, and missing-field fallback behavior

## Implementation Breakdown

### 1. Strategic run metadata

- add act-level strategic metadata such as `act_index`, `boss_id`, and `boss_name` where available
- expose the same fields in compact state and agent-view payloads
- record explicit `null` or `unknown` values when the game screen does not currently expose them

### 2. Full map contract

- expose a graph-shaped map structure with node ids, node types, coordinates, states, and outgoing edges
- expose the current node and the currently reachable frontier
- preserve the existing `available_nodes` contract so simple callers still work

### 3. Typed model and summary integration

- add typed payloads for boss and map-graph fields
- extend `build_state_summary` to capture boss identity, frontier composition, and graph-level counts
- make the strategic fields available to predictors, planners, and benchmark summaries

### 4. Replay and logging compatibility

- version the trajectory schema carefully so old artifacts remain readable
- keep replay and compare tooling tolerant of sessions recorded before the new fields existed
- surface strategic-state availability in build warnings or schema metadata when needed

### 5. Tests

- add fixture payloads for missing, partial, and complete strategic state
- test trajectory summary generation with the new boss and map graph fields
- test runtime parsing and schema migration expectations

## Deliverables

- richer `STS2-Agent` strategic state payload
- typed env models for boss and map graph
- updated trajectory summary contract
- compatibility tests for old and new state payloads
