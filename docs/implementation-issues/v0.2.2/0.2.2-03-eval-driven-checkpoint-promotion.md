# Issue 0.2.2-03

## Title

Eval-driven checkpoint promotion

## Status

Completed on the current branch.

## Objective

Let schedule runs choose the next checkpoint from explicit latest-vs-best evaluation artifacts instead of relying only on training-time reward snapshots.

## Problem

`v0.2.1` schedules could resume from `latest` or `best`, but they could not resolve that choice with actual evaluation. The result was a blind tradeoff between recency and reward snapshots with no persisted promotion rationale.

## In Scope

- add `checkpoint_source=best_eval`
- compare `latest` vs `best` after each session using the checkpoint comparison harness
- persist promotion artifacts, selected checkpoint label, and fallback reason into schedule outputs
- expose the new controls through the CLI and wrapper script

## Out Of Scope

- multi-checkpoint tournaments
- asynchronous evaluator farms
- policy distillation or predictor-based promotion

## Dependencies

- `0.2.2-02` Replay and compare artifact hardening

## Affected Areas

- `src/sts2_rl/train/scheduler.py`
- `src/sts2_rl/train/__init__.py`
- `src/sts2_rl/cli.py`
- `scripts/run-local-single.ps1`
- `tests/test_train_scheduler.py`

## Acceptance Criteria

- schedules accept `checkpoint_source=best_eval`
- latest-vs-best evaluation artifacts are written under the schedule output tree
- session summaries expose the selected checkpoint label, selection mode, and fallback metadata
- tests cover both successful promotion and fallback behavior

## Verification

- automated scheduler tests for `best_eval`
- local schedule smoke can emit `promotions/` artifacts and resume from the selected checkpoint

## Delivered

- schedule runs now support `checkpoint_source=best_eval` with configurable eval budgets, normalization target, and fallback choice.
- session summaries now embed `checkpoint_selection` payloads with comparison paths, delta metrics, and error/fallback details.
- the single-instance wrapper exposes schedule-side `best_eval` controls and a `normalize` helper command.
