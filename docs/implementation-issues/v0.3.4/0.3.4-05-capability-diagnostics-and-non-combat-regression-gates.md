# Issue 0.3.4-05

## Status

Complete.

## Title

Capability diagnostics and non-combat regression gates

## Objective

Turn non-combat capability gaps into measurable artifacts and release gates so unsupported descriptors, semantic ambiguity, and no-action stalls become actionable regression signals instead of ad hoc log inspection.

## Problem

Today the repo can emit `build_warnings` and stop runs with `policy_no_action_timeout`, but those signals are still too coarse for promotion and too weak for ownership attribution. Without a benchmark-grade diagnostic layer, future contract and policy work will regress silently or create arguments about whether a failure belongs to this repo or `STS2-Agent`.

## Primary Owner

This repo

## Dependencies

- `0.3.4-01` runtime semantic contract and build-lineage hardening
- `0.3.4-02` transactional selection planner and execution
- `0.3.4-04` quest, special-rest, and non-combat action-space completion

## Affected Areas

- `src/sts2_rl/collect/runner.py`
- `src/sts2_rl/train/runner.py`
- `src/sts2_rl/train/divergence.py`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## In Scope

- richer diagnostics for unsupported descriptors, semantic ambiguity, and non-combat timeouts
- benchmark or replay-suite gates focused on non-combat capability coverage
- explicit ownership attribution in summaries and failure reports
- promotion guardrails that prevent silent regression of supported non-combat flows

## Out Of Scope

- replacing existing combat benchmark suites
- promotion decisions driven only by public-data metrics
- broad observability work unrelated to non-combat support

## Acceptance Criteria

- session and benchmark artifacts distinguish at least `runtime contract gap`, `repo action-space gap`, `repo policy gap`, and `unexpected runtime divergence`
- unsupported descriptor counts, no-action timeouts, and ambiguous-semantic blocks are summarized in machine-readable outputs
- benchmark or replay gates can fail a candidate when a previously supported non-combat path regresses
- reports make it easy to identify which issues should be fixed in this repo versus `STS2-Agent`
- tests cover diagnostic classification, artifact rendering, and at least one non-combat regression gate

## Implementation Breakdown

### 1. Diagnostic taxonomy

- define a durable taxonomy for non-combat capability failures
- thread the taxonomy through runtime summaries, runner stop reasons, and benchmark outputs

### 2. Artifact and CLI reporting

- add machine-readable summaries and human-readable CLI views for non-combat capability failures
- preserve links back to the originating screen, descriptor, and ownership bucket

### 3. Regression gates

- add benchmark or replay-suite checks for supported non-combat paths
- keep the gates strict enough to catch regressions without blocking on intentionally unsupported flows

### 4. Documentation

- document how to use the new diagnostics during rollout, promotion, and cross-repo debugging

## Deliverables

- non-combat capability diagnostic taxonomy and artifacts
- regression gates for supported non-combat flows
- tests and documentation

## Notes

- this issue is the closure step for the patch line because it makes the earlier contract and planner work operationally enforceable

## Implementation Notes

- trajectory session summaries now persist `non_combat_capability` with ownership buckets, reason histograms, regression keys, and per-diagnostic detail rows
- no-action timeout paths now record structured capability diagnostics instead of relying on raw `policy_no_action_timeout:*` stop strings alone
- benchmark case and suite summaries now aggregate non-combat capability diagnostics for eval and compare runs
- strategic promotion gates now fail on new non-combat capability regressions and can optionally cap total candidate-side capability issues
- CLI reporting now includes `eval capability-summary` for session, case, and suite artifacts
