# Issue 0.3.4-01

## Status

Complete on the current branch.

## Title

Runtime semantic contract and build-lineage hardening

## Objective

Make non-combat runtime semantics and build identity explicit in the `STS2-Agent` HTTP contract so this repo can stop inferring selection mode, source context, and patch lineage from prompt text and fragile screen history.

## Problem

The current repo can see `selection.min_select`, `selection.max_select`, `selection.selected_count`, and card options, but it still has to infer critical meaning from `selection.prompt`, `selection.kind`, and local screen history. That is enough for narrow heuristics such as `shop_remove`, but not enough for reliable transactional selection support, generalized strategic training, or strict attribution of runtime failures. At the same time, Early Access patch churn means public and runtime data need stronger build and branch identity than a loose `game_version` string passed around informally.

## Primary Owner

`STS2-Agent`

## In Scope

- extend the runtime HTTP state contract with explicit non-combat semantic fields for selection and closely related follow-up screens
- expose build identity fields that are strong enough for build-tight dataset and benchmark matching
- add repo-side typed model support, validation, and artifact persistence for the new fields
- define clear failure behavior when the runtime cannot provide required semantics yet

## Out Of Scope

- implementing repo-side transactional multi-card planning in the same issue
- broad trainer or benchmark changes beyond consuming the new runtime metadata
- treating prompt parsing as an equally preferred long-term fallback once explicit fields exist

## Dependencies

- existing `STS2-Agent` HTTP health and state endpoints
- current repo-side typed client, wrapper, and summary infrastructure

## Acceptance Criteria

- the runtime contract exposes explicit build identity such as build id, game version, and beta-branch or channel metadata in a stable typed form
- selection payloads expose explicit semantic mode and source context instead of requiring repo-side prompt parsing for core meaning
- selection payloads expose enough transaction context to distinguish one-card and multi-card flows without reconstructing intent from local history alone
- repo-side wrappers persist the new fields into observations, summaries, and diagnostics
- ambiguous or missing semantics are surfaced as explicit capability gaps rather than silently guessed
- tests cover typed model validation, missing-field behavior, and runtime-metadata persistence

## Implementation Breakdown

### 1. Runtime contract design

- define the minimum semantic fields needed for non-combat control, including selection mode, source context, and transaction or bundle context
- define the minimum build-lineage fields needed for data filtering and benchmark attribution
- document how unknown or unavailable semantics are represented without overloading free-form prompt strings

### 2. Repo-side typed ingestion

- extend repo-side payload models and runtime snapshots to ingest the new semantic and build fields
- preserve backward-compatible validation only where the runtime has not yet been upgraded, with explicit warnings instead of silent downgrade
- thread the new metadata into observation summaries and artifact lineage

### 3. Capability reporting

- add structured warnings when a runtime screen still lacks required semantics for supported policy paths
- separate `repo missing policy logic` from `runtime missing contract data` in diagnostics

### 4. Documentation

- document the ownership boundary between this repo and `STS2-Agent`
- document which non-combat features remain blocked until the runtime contract lands

## Deliverables

- explicit runtime semantic and build-lineage contract specification
- repo-side typed ingestion and diagnostics for the new fields
- tests and documentation

## Delivered On The Current Branch

- `STS2-Agent` `/health` and `/state` now expose explicit runtime build-lineage fields: `build_id`, `branch`, `content_channel`, `commit`, `build_date`, and `main_assembly_hash`
- `/state.selection` now exposes explicit `selection_family`, `semantic_mode`, `source_type`, `source_room_type`, prompt localization keys, required and remaining counts, and multi-select capability flags
- non-combat deck-selection transaction metadata now reports selected-count and confirmation readiness instead of leaving those fields combat-hand-only
- repo-side typed payload models now ingest the expanded runtime contract and persist it into state summaries
- strategic runtime context and runtime strategic-finetune extraction now consume runtime build ids from state payloads when explicit config overrides are absent
- selection policy logic now requires explicit runtime semantics for deck-selection handling and uses explicit `source_type == "shop"` gating for `shop_remove` strategic guidance
- regression coverage now validates expanded health parsing, expanded state parsing, explicit-semantic selection policy behavior, and strategic runtime extraction under the new contract

## Notes

- this issue intentionally treats `STS2-Agent` as the authoritative source of non-combat semantics
- prompt text may remain as a temporary fallback only until the explicit runtime contract is available, not as the steady-state design
