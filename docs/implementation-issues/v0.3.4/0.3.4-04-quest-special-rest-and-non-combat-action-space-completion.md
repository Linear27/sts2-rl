# Issue 0.3.4-04

## Status

Complete on the current branch.

## Title

Quest, special-rest, and non-combat action-space completion

## Objective

Close the remaining high-value non-combat action-space gaps by making quest-linked screens, special rest-site interactions, and other currently under-specified flows first-class supported paths instead of falling through to unsupported descriptors or fragile generic handling.

## Problem

Even after selection semantics are hardened, some non-combat failures will still come from screens that are not cleanly represented in the runtime action contract or not mapped by this repo's candidate builder and policy pack. Quest-linked follow-up screens, special rest-site options, and similar Early Access additions are the likely places where the current stack still risks `Unsupported action descriptor` warnings or `policy_no_action_timeout` stops.

## Primary Owner

Shared: `STS2-Agent` for runtime descriptors and state coverage, this repo for candidate expansion and policy support

## Dependencies

- `0.3.4-01` runtime semantic contract and build-lineage hardening
- `0.3.4-02` transactional selection planner and execution

## Affected Areas

- `src/sts2_rl/env/candidate_actions.py`
- `src/sts2_rl/env/models.py`
- `src/sts2_rl/collect/policy.py`
- `src/sts2_rl/collect/runner.py`
- `tests/`
- `docs/`

## In Scope

- identify the targeted non-combat screens and descriptors that are currently under-specified or unsupported
- extend runtime descriptors or payload fields where `STS2-Agent` is the bottleneck
- extend candidate-action builders and policy handlers where this repo is the bottleneck
- preserve explicit control-scope semantics so unsupported multiplayer-only or remote-player flows do not masquerade as local-player actions

## Out Of Scope

- full co-op strategy planning
- combat action-space redesign
- generic "support everything" work without a scoped target-screen list

## Acceptance Criteria

- the targeted quest, special-rest, and adjacent non-combat screens have explicit runtime descriptors and state fields where needed
- repo-side candidate expansion recognizes the targeted descriptors without emitting unsupported-action warnings for supported screens
- policy-pack support exists for the targeted local-player flows, or the repo emits explicit structured unsupported-capability diagnostics
- control-scope handling prevents remote-player or multiplayer-only actions from being treated as local-player policy opportunities
- tests cover runtime-to-candidate expansion and policy completion for the targeted screens

## Implemented Target Inventory

- quest-linked event follow-up reward and card-selection screens
  owner: shared
  implementation: `STS2-Agent` now persists source action, event id, event option index, and event option key/title into `reward` and `selection` payloads after `choose_event_option`; this repo now consumes those fields in typed models, state summaries, policy routing, and runtime strategic extraction
- special rest-site follow-up selection screens
  owner: shared
  implementation: `STS2-Agent` now persists source action plus rest option id/index/title into `selection` and `reward` payloads after `choose_rest_option`; this repo now preserves that lineage in summaries and training/runtime extraction
- relabeled or special rest options on the rest screen itself
  owner: this repo
  implementation: rest-policy scoring now treats canonical `option_id` values as first-class semantics, so upgrade/remove/dig-like rest actions do not depend on UI-title matching alone
- reward-room source misclassification after non-combat follow-ups
  owner: this repo
  implementation: reward strategic routing now uses explicit runtime `reward.source_type` instead of defaulting non-combat follow-up rewards to monster rewards

## Owner Classification Notes

- no separate `quest` screen or dedicated `quest_*` action descriptor is currently exposed by `STS2-Agent`; on the current branch quest flows are represented as event-driven follow-up lineage rather than a distinct runtime screen family
- this issue intentionally does not invent a synthetic quest taxonomy inside this repo; it consumes authoritative runtime event ids and option keys instead of prompt-text heuristics
