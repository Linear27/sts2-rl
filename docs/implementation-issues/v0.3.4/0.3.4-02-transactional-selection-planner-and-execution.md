# Issue 0.3.4-02

## Status

Complete on the current branch.

## Title

Transactional selection planner and execution

## Objective

Upgrade non-combat selection handling from per-click greedy choice into a transactional planner that can complete multi-card remove, upgrade, pick, and transform flows as one coherent decision bundle.

## Problem

The current policy can click `select_deck_card` and later `confirm_selection`, but it still treats each card as an isolated score instead of one transaction with a required count, ordered picks, confirmation, and recovery. That gap is most visible in multi-card selection scenarios and event-driven deck operations, where the agent may be able to click a legal card but still lacks a reliable semantic plan for finishing the whole bundle.

## Primary Owner

This repo

## Dependencies

- `0.3.4-01` runtime semantic contract and build-lineage hardening
- existing `collect/policy.py`, `collect/runner.py`, and observation summary plumbing

## Affected Areas

- `src/sts2_rl/collect/policy.py`
- `src/sts2_rl/collect/runner.py`
- `src/sts2_rl/data/trajectory.py`
- `tests/`
- `docs/`

## In Scope

- transactional planning for supported multi-card selection bundles
- explicit execution state that tracks remaining selections, already-selected cards, and confirmation readiness
- separate handling for pick, remove, upgrade, and transform semantics when the runtime contract provides them
- trace metadata and summaries that preserve the selection transaction instead of only the last clicked card

## Out Of Scope

- introducing broad learning-based non-combat planning in the same issue
- broad multiplayer strategy logic
- strategic dataset or trainer changes beyond what is required for runtime trace fidelity

## Acceptance Criteria

- policy decisions for multi-card selection flows are planned and executed as one transaction rather than a sequence of unrelated single-card picks
- supported transaction types include at least multi-card remove, multi-card upgrade, and multi-card pick, with transform treated separately when the runtime contract distinguishes it
- confirmation is executed only when the transaction plan is satisfied, and the planner can recover cleanly if the runtime returns an unexpected intermediate state
- decision traces and run summaries persist transaction metadata such as required count, selected count, semantic mode, and chosen bundle members
- tests cover multi-card completion, confirmation timing, and recovery from partial or stale runtime state

## Implementation Breakdown

### 1. Transaction state model

- define a policy-side transaction object for selection bundles
- capture required count, semantic mode, candidate pool, chosen set, and confirmation state

### 2. Planning and ranking

- rank cards at the bundle level instead of pretending each click is a fully independent decision
- support stable tie-breaking and avoid reselecting already-chosen cards
- treat transform as distinct from remove when runtime semantics allow it

### 3. Execution and recovery

- execute the planned bundle across repeated observations until the transaction is complete
- verify each intermediate state before advancing or confirming
- fail loudly with actionable diagnostics when the runtime diverges from the expected transaction state

### 4. Artifact fidelity

- write transaction metadata into decision traces and trajectory summaries
- make downstream tooling able to distinguish a complete selection transaction from a partial click sequence

## Deliverables

- transactional selection planner and executor
- richer selection decision traces and summaries
- tests and documentation

## Delivered

- `SimplePolicy` now maintains an explicit selection transaction state keyed by run, screen, semantic mode, source, localization prompt identity, and required bundle size
- deck selection planning now ranks the full candidate set once, stores a bundle plan, advances that plan across repeated observations, and avoids reselecting already-completed bundle members
- supported transaction semantics now include `pick`, `remove`, `upgrade`, and `transform`, with transform treated as its own execution mode instead of a remove alias in traces
- transaction reconciliation now uses explicit runtime `selected_count`, `remaining_count`, and confirmation readiness to advance, replan, or fail loudly when the runtime diverges
- decision traces now emit `selection_transaction` metadata that records phase, planned members, completed members, next member, confirmation state, and recovery diagnostics
- trajectory state summaries now preserve selection confirmation fields so later dataset and artifact consumers can distinguish incomplete vs confirmable transaction states
- regression coverage now includes multi-step bundle completion, bundle replanning after divergence, transform-mode transaction routing, and summary-field persistence

## Notes

- this issue is the main repo-side answer to the currently visible "choose multiple cards" gap
- if `STS2-Agent` cannot yet provide explicit semantics, this issue should remain blocked on those paths rather than doubling down on prompt heuristics
