# Issue 0.3.4-03

## Status

Complete on the current branch.

## Title

Strategic selection-fidelity datasets, training, and evaluation

## Objective

Extend the current strategic-data stack so generalized selection semantics, transactional selection traces, and strict build matching become first-class training and evaluation inputs rather than a narrow `shop_remove` exception.

## Problem

`v0.3.3` made public strategic decisions and runtime-side strategic fine-tuning operational, but the runtime bridge is still intentionally narrow. Today the repo can confidently train on reward picks, shop buys, event choices, rest-site actions, and a strict shop-remove path. That leaves a real fidelity gap between what the live policy needs to execute and what the strategic models can learn or evaluate, especially for generalized selection flows and beta-version drift.

## Primary Owner

This repo

## Dependencies

- `0.3.4-01` runtime semantic contract and build-lineage hardening
- `0.3.4-02` transactional selection planner and execution
- existing `v0.3.3` strategic dataset and trainer stack

## Affected Areas

- `src/sts2_rl/train/strategic_pretrain.py`
- `src/sts2_rl/train/strategic_finetune.py`
- `src/sts2_rl/train/policy_checkpoint.py`
- `src/sts2_rl/collect/strategic_runtime.py`
- `configs/`
- `tests/`
- `docs/`

## In Scope

- new strategic decision typing for supported selection semantics beyond the current shop-remove-only bridge
- dataset and training summary lineage that preserves build, branch, and semantic fidelity
- strategic runtime guidance that can consume the stronger selection decision families when support quality is explicit
- benchmark and evaluation reporting for the expanded strategic selection coverage

## Out Of Scope

- treating public strategic examples as control-equivalent step supervision
- broad offline-RL transition generation from public selection traces
- policy-side execution changes already owned by `0.3.4-02`

## Acceptance Criteria

- strategic runtime extraction can represent supported selection decisions with explicit semantics and support-quality metadata instead of collapsing them into a generic or shop-only path
- strategic datasets can filter and summarize by build id, beta branch, and decision subtype for the newly supported selection families
- strategic pretraining and fine-tuning checkpoints preserve lineage for the stronger selection semantics without mixing incompatible builds
- live strategic runtime guidance can score supported selection candidates using the expanded decision families
- tests cover dataset materialization, training-example extraction, build-match guardrails, and live guidance integration for the new selection semantics

## Implementation Breakdown

### 1. Decision taxonomy expansion

- define new supported strategic decision types for generalized selection semantics
- represent support quality explicitly so full-candidate and weaker reconstruction paths remain distinguishable

### 2. Dataset and training updates

- extend public and runtime strategic loaders to emit the new selection decision families
- preserve build and branch lineage in summaries, metrics, and checkpoint metadata
- reject incompatible build mixes by default

### 3. Runtime guidance integration

- wire the expanded decision families into live strategic guidance hooks
- keep unsupported or weakly reconstructed cases explicitly out of authoritative paths

### 4. Evaluation and reporting

- add benchmark and held-out metrics for the new selection decision families
- make warm-start and live-guidance reports attribute gains or failures to the stronger selection semantics

## Deliverables

- expanded strategic decision taxonomy for selection semantics
- dataset, trainer, and evaluation updates with strict build lineage
- tests and documentation

## Delivered

- strategic decision typing now includes generalized runtime selection families such as `selection_pick`, `selection_remove`, `selection_upgrade`, and `selection_transform`, with runtime guidance no longer hard-wired to `shop_remove`
- runtime strategic extraction now emits selection decisions directly from explicit runtime semantics and transaction metadata, including selection phase, required count, remaining count, multi-select capability, and recovery count
- public strategic dataset materialization now emits generalized `selection_remove` records instead of a shop-only remove subtype, while preserving explicit `source_type` and support quality
- public strategic decision records and dataset summaries now preserve richer build lineage fields and histograms, including `game_version`, `branch`, and `content_channel` when available
- strategic context features now include selection semantic metadata plus richer build-lineage features, so pretraining, fine-tuning, and live runtime guidance consume the same stronger signal surface
- mixed strategic fine-tuning now persists runtime/public dataset decision histograms and build-lineage histograms in summaries and checkpoints, and rejects disjoint runtime/public or warm-start/dataset lineage by default
- regression coverage now includes generalized selection runtime guidance, public dataset materialization under the new selection taxonomy, runtime/public build-match guardrails, and expanded fine-tuning lineage assertions

## Notes

- this issue is about strategic-fidelity parity with the new runtime contract, not about relaxing the public-data trust boundary
