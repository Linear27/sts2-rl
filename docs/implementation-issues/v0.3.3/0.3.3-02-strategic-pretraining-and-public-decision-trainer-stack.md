# Issue 0.3.3-02

## Status

Complete on the current branch.

## Title

Strategic pretraining and public-decision trainer stack

## Objective

Train strategic policy and value modules from `public_strategic_decisions` so the repo can use large public corpora for initialization without confusing partial strategic supervision with full runtime control imitation.

## Problem

Once `public_strategic_decisions` exists, the repo still needs a training path that respects its weaker supervision shape. Reusing the current behavior-cloning trainer directly would encourage invalid comparisons between strategic decision examples and runtime-faithful step labels. The repo needs a separate trainer stack for strategic ranking, masked choice prediction, and value-style objectives that can exploit public data safely and produce reusable warm-start checkpoints.

## In Scope

- a strategic pretraining trainer for public strategic decisions
- masked candidate-choice losses for reward, shop, event, and rest decisions, plus explicit handling for `chosen_only` weak-supervision records
- optional auxiliary value or outcome heads for long-horizon strategic context
- confidence-aware sample weighting, decision-type weighting, and version filtering
- checkpoint, metrics, and summary contracts compatible with later fine-tuning

## Out Of Scope

- direct combat action imitation from public runs
- offline RL objectives over public-run-derived pseudo transitions
- generic large-sequence pretraining infrastructure
- replacing the existing BC trainer for runtime trajectory datasets

## Dependencies

- `0.3.3-01` public strategic decision dataset contract
- current model registry, artifact layout, and benchmark reporting conventions
- existing predictor and policy-pack feature extraction infrastructure where reusable

## Affected Areas

- `src/sts2_rl/train/`
- `src/sts2_rl/data/`
- `src/sts2_rl/eval/`
- `src/sts2_rl/cli.py`
- `tests/`

## Acceptance Criteria

- the repo can train a strategic checkpoint directly from a `public_strategic_decisions` dataset with no one-off conversion scripts
- training supports full-candidate ranking objectives for strong labels plus explicit positive-only objectives for `chosen_only` weak-supervision records
- summaries report dataset lineage, build coverage, confidence-weight distributions, per-decision-type metrics, held-out ranking or top-k accuracy, and auxiliary value metrics
- checkpoints persist enough metadata to warm-start later strategic fine-tuning or compatible runtime-side modules
- tests cover dataset ingestion, weighting, checkpoint metadata, held-out metrics, and CLI execution

## Implementation Breakdown

### 1. Training-ready dataset adapter

- define a trainer view over `public_strategic_decisions` with deterministic ordering and split handling
- preserve candidate sets and chosen labels exactly as materialized by the dataset builder
- support sample weighting by reconstruction confidence, run outcome, source, and decision type

### 2. Strategic model heads

- add stage-aware heads for reward, shop, event, and rest decisions
- allow optional shared encoders or decision-family-specific heads when label spaces differ materially
- keep combat control heads out of this trainer stack

### 3. Objectives and metrics

- train with candidate-choice softmax objectives over `full_candidates` records
- train with explicit positive-only objectives over `chosen_only` records instead of fabricating unsupported negatives
- add auxiliary value heads where outcome context improves representation quality
- report per-domain accuracy, top-k recall, candidate-support coverage, and value-log-loss style diagnostics

### 4. Checkpoint and artifact contract

- persist dataset lineage, source manifests, build filters, and feature schema metadata with each run
- mark checkpoints explicitly as `strategic_pretrain` artifacts so later consumers cannot confuse them with runtime BC or offline RL checkpoints
- emit benchmark-compatible summaries where comparison is structurally meaningful

### 5. Docs and guardrails

- document when strategic pretrained checkpoints may warm-start later trainers
- fail clearly when datasets mix incompatible builds or unsupported decision families

## Deliverables

- strategic pretraining trainer and CLI commands
- strategic checkpoint artifact contract
- metrics, tests, and documentation

## Delivered On The Current Branch

- `strategic_pretrain` is now a first-class training path with its own config, checkpoint schema, summary contract, and CLI command
- the trainer consumes `public_strategic_decisions` datasets directly through the existing manifest-built dataset flow
- `full_candidates` examples train through candidate-choice ranking objectives, while `chosen_only` examples train through explicit positive-only weak-supervision objectives
- the trainer also learns auxiliary value heads from run outcomes and reports held-out ranking, positive-only, and value metrics per decision type
- checkpoints persist dataset lineage, feature inventory hashes, decision-type coverage, and training config metadata for later warm-start usage
- regression tests cover trainer artifacts, learned ranking behavior, summary metrics, and CLI execution

## Reference-Driven Decisions

- public runs provide strong strategic supervision at route, reward, shop, and event granularity, but do not provide the runtime-faithful control stream expected by BC and offline RL trainers
- passive-data and imitation-from-observation literature supports using large observational corpora for initialization, representation shaping, and downstream acceleration without treating them as equivalent to action-complete demonstrations
- the repo's existing public-source work already separates aggregate-card priors from run-derived strategic signals, so training should preserve the same separation instead of collapsing everything into a single policy-supervision path
