# Issue 0.3.3-01

## Status

Complete on the current branch.

## Title

Public strategic decision dataset contract

## Objective

Define a repo-native `public_strategic_decisions` dataset family that turns normalized public runs into version-scoped, provenance-rich strategic training examples without pretending those examples are full step-level trajectories.

## Problem

`v0.3.2` makes public data useful for priors, predictors, and benchmark diagnostics, but the repo still has no training-ready artifact family for public-run-derived decisions. If those examples are pushed into `trajectory_steps` or `offline_rl_transitions`, the contract becomes misleading and future trainers will over-trust partial supervision as if it were runtime-faithful control data. The repo needs an explicit dataset family for supported strategic decisions with strict build matching, reconstruction confidence, dedupe rules, and candidate-set coverage.

## In Scope

- a new `public_strategic_decisions` dataset kind and artifact contract
- builders that derive supported decision examples from normalized public-run artifacts
- strict version, build, and beta-branch filtering
- explicit candidate sets, selected action labels, confidence fields, and outcome metadata
- manifest / summary / CSV or JSONL exports plus tests and docs

## Out Of Scope

- combat micro-action reconstruction
- adding public runs to `trajectory_steps` or `offline_rl_transitions`
- live web fetches during dataset build
- broad multi-source fusion before source-specific reconstruction quality is understood

## Dependencies

- `0.3.2-02` public run archive sync
- `0.3.2-03` normalized public-run strategic summaries
- `0.3.2-04` provenance and freshness diagnostics in existing reports
- current dataset registry and manifest builder conventions

## Affected Areas

- `src/sts2_rl/data/`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## Acceptance Criteria

- the repo can materialize a deterministic `public_strategic_decisions` dataset from local normalized public-run artifacts
- each example records `decision_type`, `chosen_action`, `reconstruction_confidence`, `build_id`, `source_name`, and `snapshot_date`, with `candidate_actions` present only when the source supports faithful reconstruction
- the builder supports the currently credible decision types: `reward_card_pick`, `shop_buy`, `shop_remove`, `event_choice`, and `rest_site_action`
- dataset summaries expose source coverage, build coverage, decision-type counts, confidence histograms, support-quality splits, and dedupe statistics
- tests cover version filtering, candidate-set reconstruction, chosen-only weak-supervision records, and manifest reproducibility

## Implementation Breakdown

### 1. Dataset family and record schema

- add a new dataset kind instead of reusing `trajectory_steps`
- define a typed record with decision metadata, candidate actions, chosen label, run context, and source lineage
- persist explicit reconstruction-confidence and support-quality fields so downstream trainers can weight samples safely

### 2. Supported decision extractors

- derive examples only from decision points that public runs represent credibly
- keep separate extractors for reward, shop, event, and rest decisions so each can evolve independently
- distinguish `full_candidates` examples from `chosen_only` weak-supervision examples instead of silently inventing candidate sets

### 3. Version and source controls

- require explicit build filters and default to strict version matching
- carry source usage constraints and attribution metadata into source manifests
- support minimum freshness and minimum sample-size guards when requested

### 4. Dedupe and leakage controls

- dedupe by stable run or decision lineage keys
- expose player/source partitions where available so evaluation splits do not leak near-duplicate examples
- keep train or validation split grouping deterministic at the run and decision-family levels

### 5. Inspection and documentation

- add summary and inspection CLI flows similar to the existing dataset tooling
- document why `public_strategic_decisions` is a strategic dataset family rather than a control-trajectory family

## Deliverables

- `public_strategic_decisions` artifact contract and dataset builder
- CLI and summary wiring
- tests and issue / roadmap documentation

## Delivered On The Current Branch

- `public_strategic_decisions` is now a first-class dataset kind with explicit `public_run_normalized` source support
- the dataset builder materializes deterministic strategic decision records from local normalized public-run artifacts
- reward and shop buy decisions preserve full candidate sets, while shop remove, event choice, and rest-site actions are marked explicitly as `chosen_only`
- dataset summaries now expose decision-type, support-quality, build, source, and confidence diagnostics for the new dataset family
- regression tests cover source discovery, manifest validation, build filtering, confidence filtering, candidate reconstruction, and weak-supervision extraction

## Reference-Driven Decisions

- public runs are already normalized locally into route, reward, shop, event, relic, and encounter summaries, which makes strategic decision extraction feasible without live scraping
- Mega Crit's current early-access cadence and public-beta usage make strict build filters mandatory because cross-patch labels will drift quickly
- public ecosystem APIs expose large public corpora, but their fields and terms are better aligned with local snapshot ingestion than with direct online training dependencies
- offline RL literature assumes transition-level datasets, while passive-data and observation-only work supports using partial public signals as weaker strategic supervision rather than as full control labels
