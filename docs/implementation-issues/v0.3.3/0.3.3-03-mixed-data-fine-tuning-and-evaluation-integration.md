# Issue 0.3.3-03

## Status

Complete on the current branch.

## Title

Mixed-data fine-tuning and evaluation integration

## Objective

Warm-start compatible runtime trainers from strategic public-data checkpoints and evaluate the resulting mixed-data policies without ever treating public strategic examples as control-equivalent trajectory supervision.

## Problem

Strategic pretraining is only valuable if it improves later runtime-side learning and evaluation. Without an explicit fine-tuning and comparison contract, the repo would either leave public-data checkpoints disconnected from the main training stack or, worse, merge heterogeneous public strategic samples directly into runtime trajectory corpora. The repo needs a disciplined bridge that allows warm-start and comparative evaluation while preserving the stronger trust boundary of local runtime data.

## In Scope

- warm-start hooks from `strategic_pretrain` checkpoints into compatible runtime-side trainers
- mixed-data training schedules that keep public strategic examples separate from runtime trajectory batches
- evaluation and benchmark reporting that compare pretrain vs no-pretrain runs with explicit dataset lineage
- guardrails that prevent unsupported checkpoint or dataset mixing

## Out Of Scope

- converting `public_strategic_decisions` into `trajectory_steps`
- mixing public examples directly into offline RL transition datasets
- live online learning from third-party public APIs
- automatic checkpoint promotion based only on public-data-backed metrics

## Dependencies

- `0.3.3-02` strategic pretraining trainer stack
- existing behavior-cloning, offline RL, predictor, benchmark, and experiment-registry infrastructure

## Affected Areas

- `src/sts2_rl/train/`
- `src/sts2_rl/eval/`
- `src/sts2_rl/registry/`
- `src/sts2_rl/cli.py`
- `tests/`
- `configs/`
- `docs/`

## Acceptance Criteria

- runtime-side trainers can optionally initialize compatible weights from a `strategic_pretrain` checkpoint through explicit config
- training summaries and registry metadata preserve both the public-data pretraining lineage and the runtime fine-tuning lineage
- benchmark and compare reports can attribute gains or regressions to warm-start usage instead of collapsing them into generic checkpoint comparisons
- guardrails reject incompatible feature schemas, incompatible builds, or unsupported head mappings
- tests cover checkpoint import, config validation, lineage propagation, and benchmark-summary rendering

## Implementation Breakdown

### 1. Warm-start contract

- define which encoder and head components can transfer from strategic checkpoints into runtime trainers
- require explicit opt-in and compatible schema checks
- keep unsupported heads frozen out rather than silently mapping them

### 2. Mixed-data scheduling

- allow curriculum or two-stage training where public strategic pretraining happens first and runtime fine-tuning happens second
- keep heterogeneous datasets separate in manifests, summaries, and dataloaders
- expose weighting and freeze or unfreeze controls for transferred components

### 3. Evaluation and reporting

- add benchmark and registry metadata for pretrain lineage, transferred modules, and fine-tune schedule
- compare warm-start vs cold-start runs using the existing benchmark framework plus strategic-alignment diagnostics where relevant
- keep public-data metrics informative but non-authoritative for final promotion decisions

### 4. Promotion guardrails

- require final benchmark decisions to remain anchored to runtime-collected evals
- document that public-data-backed strategic metrics are supportive evidence, not sufficient promotion evidence

### 5. Tests and docs

- test explicit warm-start mapping, invalid-schema rejection, and summary lineage propagation
- document recommended training schedules for new builds and rapidly changing beta patches

## Deliverables

- warm-start and mixed-data fine-tuning integration
- lineage-rich reporting and registry support
- tests and documentation

## Delivered On The Current Branch

- added `train strategic-finetune` as a first-class runtime-side trainer instead of forcing public strategic checkpoints into behavior-cloning head mappings
- strategic fine-tuning now extracts only explicit runtime strategic decisions from local `trajectory_steps`, including reward-card picks or skips, shop buys, event options, rest-site actions, and shop-removal selections gated by explicit `remove_card_at_shop` context
- mixed schedules now keep runtime and public strategic corpora separate through dataset resolution, per-source epoch scheduling, per-source metrics, and summary lineage
- warm-start now supports direct transfer from `strategic_pretrain` or prior `strategic_finetune` checkpoints with explicit feature-schema checks, transferred-module reporting, and optional freeze windows for transferred ranking/value heads
- summary artifacts now preserve runtime lineage, public lineage, warm-start references, transferred-module metadata, and source-specific held-out metrics instead of collapsing everything into one generic training report
- experiment registry inspection now recognizes both `strategic_pretrain` and `strategic_finetune` training artifacts and preserves warm-start plus mixed-dataset lineage in registered entries
- regression coverage now includes runtime strategic sample extraction, mixed-source fine-tuning, CLI wiring, and registry registration for strategic fine-tuning artifacts

## Notes

- the runtime bridge remains strict: public strategic data is still not converted into `trajectory_steps` and is never mixed into offline-RL transition corpora
- `shop_remove` extraction only activates when the runtime log shows an explicit shop-removal trigger followed by a selection step with remove semantics; ambiguous deck-selection steps are still excluded
- final promotion authority is still expected to come from runtime-grounded evaluation, while public-source metrics remain initialization and analysis evidence

## Reference-Driven Decisions

- large public corpora are most valuable as initialization when the final control objective still depends on higher-fidelity local data
- rapid early-access patch drift makes continuous full retraining expensive, so strategic warm-starts can reduce sample and time costs while still preserving build-specific runtime validation
- the repo already treats public data as local snapshot artifacts with explicit provenance, which should extend naturally into mixed-data experiment lineage
