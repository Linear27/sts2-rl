# Issue 0.3.0-02

## Status

Completed on 2026-04-13.

## Title

Shadow rollout and planner-search harness

## Objective

Turn stored `shadow_combat_encounters` datasets into an executable offline research loop that can replay encounter-start snapshots through repo-native policy packs and compare candidate policies against baselines without launching the live runtime.

## Problem

`0.3.0-01` gave the repo a stable encounter-start artifact, but the repo still could not do anything operational with it. Researchers could inspect `encounters.jsonl`, yet they still lacked a first-class command that says: "reconstruct the legal action set, run an existing policy pack, measure whether it agrees with the logged action trace, and compare two policies on the same snapshots." Without that harness, shadow work would collapse back into notebooks or one-off scripts.

## In Scope

- repo-native shadow evaluation entrypoints
- stored-encounter reconstruction into `StepObservation`
- policy-pack execution over shadow combat snapshots
- baseline vs candidate comparison artifacts
- CLI summaries and tests for offline shadow runs

## Out Of Scope

- a full combat simulator
- local state-transition learning or roll-forward world dynamics
- replacing live benchmark runs with shadow-only metrics

## Dependencies

- `0.3.0-01` shadow encounter dataset contract
- existing policy-pack decision flow in `src/sts2_rl/collect/policy.py`
- existing predictor-runtime integration used by live eval and benchmark commands

## Affected Areas

- `src/sts2_rl/shadow/`
- `src/sts2_rl/cli.py`
- `tests/`
- `docs/`

## Acceptance Criteria

- the repo can evaluate a policy profile directly over stored shadow encounters with no ad hoc scripts
- evaluation reports record usable vs skipped encounters, chosen actions, trace hits, and decision metadata
- the repo can compare baseline and candidate policy profiles over identical encounter snapshots
- CLI commands expose shadow eval, compare, and summary flows with the same predictor-runtime conventions used elsewhere
- tests cover module execution, skipped-snapshot handling, CLI commands, and summary loading

## Implementation Breakdown

### 1. Shadow harness module

- add `src/sts2_rl/shadow/` as a first-class package
- reconstruct `GameStatePayload`, `AvailableActionsPayload`, and `StepObservation` from stored encounter records
- execute `build_policy_pack(...)` directly over the reconstructed snapshot

### 2. Evaluation reports

- emit per-encounter `results.jsonl`
- report first logged-action match, action-trace hit rate, decision-score stats, and skip-reason histograms

### 3. Comparison reports

- run baseline and candidate profiles over the same encounter set
- emit `comparisons.jsonl` and summarize agreement, delta match rates, and candidate-advantage signals

### 4. CLI integration

- add `sts2-rl shadow combat-eval`
- add `sts2-rl shadow combat-compare`
- add `sts2-rl shadow summary`
- reuse predictor runtime flags instead of inventing a parallel configuration format

### 5. Tests and docs

- cover a full offline encounter fixture with at least one usable and one skipped record
- cover CLI execution and summary rendering
- update roadmap, issue index, and README command examples

## Deliverables

- `src/sts2_rl/shadow/combat.py`
- `src/sts2_rl/shadow/__init__.py`
- `shadow` CLI commands and summary views
- module and CLI tests for shadow evaluation / comparison
- roadmap and README updates documenting the offline shadow loop
