# Issue 0.2.4-05

## Title

Multi-job orchestration and experiment DAGs

## Objective

Extend the current multi-instance job runner into a resumable DAG orchestrator that can execute complete experiment flows across collection, dataset build, training, evaluation, comparison, and promotion.

## Problem

`v0.2.3` makes repeated jobs reliable, but the repo still cannot express a full experiment as one managed object. In practice, meaningful work now spans several dependent stages, each with distinct artifacts, retries, and resource constraints. Without a DAG layer, those stages stay fragile and operator-heavy.

## In Scope

- DAG manifests for multi-stage experiment flows
- stage-level dependencies, inputs, outputs, retries, and resume semantics
- artifact handoff contracts between collect, dataset build, train, eval, compare, and promote stages
- resource locks and concurrency controls suitable for the current local Windows runtime
- CLI flows for validate, run, resume, inspect, and summarize orchestrated experiments

## Out Of Scope

- distributed cluster schedulers
- Kubernetes or cloud workflow engines
- fully generic workflow authoring beyond repo-relevant experiment stages

## Dependencies

- multi-instance job runner and watchdog from `0.2.3-05`
- reproducibility diagnostics from `0.2.3-06`
- registry outputs from `0.2.4-04`
- artifact contracts from `0.2.4-01` and `0.2.4-02`

## Affected Areas

- `src/sts2_rl/runtime/`
- `src/sts2_rl/train/`
- `src/sts2_rl/registry/`
- `src/sts2_rl/cli.py`
- `configs/`
- `tests/`
- `artifacts/`
- `scripts/`

## Acceptance Criteria

- an experiment DAG can be validated, executed, resumed, and summarized through repo-native commands
- failed stages preserve enough state for targeted retry instead of rerunning the whole graph
- stage artifacts and lineage are recorded in a way that registry and leaderboard tooling can consume directly
- resource conflicts across local runtime instances are prevented or surfaced explicitly
- tests cover DAG validation, topological execution, retry behavior, and artifact handoff contracts

## Implementation Breakdown

### 1. DAG manifest and execution model

- define a manifest schema for named stages, dependencies, parameters, artifacts, and retry policy
- keep the stage vocabulary focused on repo-native actions such as collect, build_dataset, train_bc, train_offline_rl, eval, compare, and promote
- persist run state so orchestration can resume after process interruption

### 2. Stage runner integration

- wrap existing job runner, dataset builder, training commands, and evaluation flows in consistent stage executors
- propagate structured outputs forward so downstream stages do not depend on manual path discovery
- preserve per-stage summaries and failure causes for post-run inspection

### 3. Resource management and retries

- add explicit resource locks for runtime instances, GPUs, and artifact destinations where needed
- support stage-level retry, cooldown, and quarantine behavior when runtime stages fail repeatedly
- separate transient runtime failure from deterministic config failure in the orchestration summary

### 4. Registry and reporting integration

- register orchestrated experiments and stage outputs into the local experiment registry as they complete
- emit end-to-end summaries that show critical path timing, retries, promoted artifacts, and skipped stages
- support concise terminal summaries plus machine-readable DAG artifacts for later tooling

### 5. Tests

- test DAG validation, dependency ordering, and cycle rejection
- test interrupted-run resume behavior and partial retry
- test resource-lock handling on synthetic multi-stage manifests
- test registry-facing artifact handoff and summary generation

## Deliverables

- experiment DAG manifest schema
- multi-stage orchestrator with resume and retry support
- resource-lock and artifact-handoff layer
- CLI support for DAG lifecycle operations
- automated tests for validation, execution, and reporting
