# Issue 0.2.4-04

## Title

Experiment registry and leaderboard outputs

## Objective

Turn artifact directories into named experiments with explicit lineage, comparable metrics, and promotion aliases so the repo can answer "what is the current best model?" without manual folder inspection.

## Problem

The repo now produces many useful artifacts, but they still mostly live as timestamped directories. That is workable for single sessions, but it does not scale to behavior cloning, offline RL, predictor retrains, and repeated benchmark sweeps across several instance configs.

## In Scope

- repo-native experiment manifests with stable identifiers, tags, aliases, and lineage
- artifact registration for policies, predictors, datasets, and benchmark reports
- leaderboard summaries across training families and benchmark suites
- promotion aliases such as `best_bc`, `best_offline_rl`, `best_predictor`, and `recommended_default`
- local-first CLI flows for register, list, compare, alias, and promote

## Out Of Scope

- mandatory external tracking servers
- cloud artifact stores
- browser-based dashboards

## Dependencies

- benchmark and summary artifacts from `0.2.3-02`
- behavior-cloning outputs from `0.2.4-01`
- offline RL outputs from `0.2.4-02`
- predictor report outputs from `0.2.4-03`

## Affected Areas

- `src/sts2_rl/registry/`
- `src/sts2_rl/cli.py`
- `configs/`
- `tests/`
- `artifacts/`
- `docs/`

## Acceptance Criteria

- experiments can be registered and queried through a stable local registry format
- registry entries preserve source dataset lineage, config fingerprints, metric snapshots, and artifact paths
- leaderboard reports can compare heuristic, BC, offline RL, predictor, and hybrid policies on shared benchmarks
- named aliases and promoted artifacts can be updated deterministically with audit-friendly history
- tests cover registry writes, compare views, alias updates, and leaderboard generation

## Implementation Breakdown

### 1. Registry schema and storage layout

- define a local registry schema for experiments, artifacts, aliases, and lineage edges
- store immutable run snapshots separately from mutable aliases and promotion pointers
- capture config fingerprints, dataset lineage, benchmark suite references, and metric summaries in the registry entry

### 2. Artifact ingestion and promotion

- add import or register commands for training runs, predictor reports, datasets, and benchmark summaries
- support explicit aliases for promoted artifacts instead of overloading raw directory names
- persist promotion history so later comparisons can explain why a default changed

### 3. Leaderboard and compare reports

- add cross-run summary builders that compare key metrics across experiment families and benchmark suites
- support filtered views by character, stage, algorithm family, runtime config, or dataset family
- write machine-readable leaderboard outputs plus concise terminal summaries

### 4. CLI and documentation

- add CLI commands for registry init, register, list, show, compare, alias, and promote
- document registry conventions and lifecycle in README and docs
- keep the implementation local-first so optional exporters can be added later without reshaping the core contract

### 5. Tests

- test registry schema validation and backward-compatible loading
- test alias updates and promotion-history persistence
- test leaderboard aggregation and filtered compare views
- test CLI-facing flows on small synthetic artifact trees

## Deliverables

- local experiment registry schema and storage layout
- artifact registration and promotion flows
- leaderboard and compare reports
- CLI support for registry lifecycle operations
- automated tests for registry and leaderboard behavior
