# Issue 0.3.5-01

## Status

Complete.

## Title

Custom-run action-space and contract-native configuration

## Objective

Make `CUSTOM_RUN` a fully consumed runtime contract in this repo by materializing the runtime's custom-run configuration descriptors in the action space and by configuring custom runs through the authoritative parameterized actions instead of repo-side incremental fallback behavior.

## Problem

`STS2-Agent` already exposes `set_custom_seed`, `set_custom_ascension`, and `set_custom_modifiers`, but this repo still treated those descriptors as unsupported in `candidate_actions` and still configured modifier bundles through repeated `toggle_custom_modifier` calls during fixed-seed preparation. That created a repo-owned capability gap even though the runtime contract already existed.

## Primary Owner

This repo

## Dependencies

- `0.3.4-04` quest, special-rest, and non-combat action-space completion
- `0.3.4-05` capability diagnostics and non-combat regression gates

## Affected Areas

- `src/sts2_rl/env/candidate_actions.py`
- `src/sts2_rl/runtime/custom_run.py`
- `tests/test_custom_run_prepare.py`
- `docs/`

## In Scope

- action-space support for `set_custom_seed`, `set_custom_ascension`, and `set_custom_modifiers`
- strict contract checks in custom-run preparation for required runtime configuration actions
- atomic modifier bundle application through `set_custom_modifiers`
- deterministic regression coverage for custom-run action ids and preparation sequences

## Out Of Scope

- changing `STS2-Agent` action names, parameter shapes, or payload semantics
- broad policy work for generic menu automation outside custom-run preparation
- introducing repo-defined fallback configuration semantics when the runtime contract is missing

## Acceptance Criteria

- `build_candidate_actions` no longer emits unsupported-descriptor warnings for runtime-supported custom-run configuration actions
- custom-run preparation configures modifiers through `set_custom_modifiers` instead of per-modifier toggles
- custom-run preparation fails loudly if a required runtime configuration descriptor or capability flag is unavailable
- regression tests cover both action-space materialization and atomic modifier-bundle preparation

## Implementation Breakdown

### 1. Action-space completion

- add candidate builders for current-seed snapshots, bounded ascension targets, and current modifier bundles
- keep candidate ids and labels deterministic so diagnostics and artifact summaries stay stable

### 2. Preparation hardening

- require runtime descriptors before sending custom-run configuration requests
- apply modifier bundles atomically and verify the resulting selected bundle after the step returns

### 3. Regression coverage

- extend `test_custom_run_prepare.py` to cover new custom-run candidate ids
- assert action-sequence changes for empty-bundle clearing and multi-modifier atomic apply

## Deliverables

- completed `CUSTOM_RUN` action-space materialization for runtime configuration descriptors
- contract-native custom-run preparation path with atomic modifier application
- targeted regression tests and roadmap issue documentation

## Implementation Notes

- `set_custom_ascension` is expanded into the bounded `0..max_ascension` discrete target space because the runtime payload exposes a finite range
- `set_custom_seed` and `set_custom_modifiers` are materialized from current runtime state snapshots so the descriptor is consumed without inventing repo-side parameter values
- custom-run preparation now treats `set_custom_modifiers` as the authoritative modifier configuration path and verifies the observed selected bundle before embark
