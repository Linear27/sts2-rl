# Issue 0.2.3-05

## Title

Multi-instance job runner and watchdog

## Objective

Make multi-instance collection and evaluation a first-class workflow with structured orchestration, retries, and health handling.

## Problem

The repo can provision and inspect instances, but unattended multi-instance work still relies too much on manual launch discipline and thin wrapper scripts. That is not enough for long-running data collection or benchmark jobs.

## In Scope

- job specs for collect, eval, replay, compare, and benchmark tasks across multiple instances
- watchdog logic for health checks, retries, cooldowns, and quarantine states
- structured job logs and summaries
- CLI entry points and local runner scripts

## Out Of Scope

- distributed cluster scheduling
- cloud autoscaling
- replacing per-instance configuration files

## Dependencies

- runtime management from `v0.2.2`
- benchmark suite definitions from `0.2.3-02`

## Affected Areas

- `src/sts2_rl/runtime/`
- `src/sts2_rl/cli.py`
- `scripts/`
- `tests/`
- `artifacts/`

## Acceptance Criteria

- a single command can orchestrate multi-instance jobs with structured output
- unhealthy instances are detected and surfaced clearly
- retries and fallback behavior are explicit in artifacts
- tests cover scheduler/watchdog decision logic

## Implementation Breakdown

### 1. Job spec layer

- define job specs for collect, eval, replay, compare, and benchmark workloads
- support per-instance assignment, concurrency limits, and output routing

### 2. Watchdog and health state model

- define health states such as healthy, degraded, cooling_down, quarantined, and failed
- react to API failures, repeated timeouts, and normalization failures consistently
- keep retry logic explicit instead of burying it in ad hoc loops

### 3. Runner orchestration

- add a repo-native runner that dispatches work across configured instances
- support graceful stop, retry, and partial completion summaries
- keep the runner compatible with the current instance config and manifest layer

### 4. Logs and artifacts

- write job-level JSON summaries with per-instance status and retry history
- make cooldown and quarantine behavior visible in logs
- keep outputs under predictable artifact roots

### 5. Tests

- test watchdog state transitions
- test retry and quarantine decisions
- test job summary generation with fake instance states

## Deliverables

- multi-instance job spec and parser
- watchdog state model
- repo-native job runner
- structured job artifacts
- automated tests for watchdog and orchestration logic
