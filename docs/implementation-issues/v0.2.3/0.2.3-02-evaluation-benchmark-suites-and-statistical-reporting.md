# Issue 0.2.3-02

## Title

Evaluation benchmark suites and statistical reporting

## Objective

Promote evaluation from ad hoc replay/compare runs into named benchmark suites with stable reports that can drive model iteration.

## Problem

The repo can already compare checkpoints, but it still lacks a durable benchmark concept. That makes it hard to answer whether one checkpoint is better across a meaningful set of scenarios instead of one-off runs.

## In Scope

- benchmark suite specs by seed, checkpoint set, floor band, and combat scope
- suite runners that aggregate replay, compare, and eval outputs
- bootstrap intervals or similar statistical summaries for core metrics
- machine-readable reports and concise human-readable summaries
- CLI entry points for suite execution

## Out Of Scope

- online dashboards
- cloud experiment tracking
- simulator-based evaluation

## Dependencies

- `0.2.3-01` Dataset registry and reproducible builds
- replay and compare infrastructure from `v0.2.2`

## Affected Areas

- `src/sts2_rl/train/`
- `src/sts2_rl/cli.py`
- `docs/`
- `tests/`
- `artifacts/`

## Acceptance Criteria

- a named benchmark suite can be executed from the CLI
- suite summaries include stable aggregate metrics and uncertainty estimates
- paired checkpoint results can be compared without reading raw logs
- tests cover aggregation and report generation

## Implementation Breakdown

### 1. Benchmark suite spec

- define a benchmark suite manifest with checkpoint inputs, runtime target, seed or scenario scope, and evaluation budgets
- support several suite modes instead of one monolithic runner:
  - single-checkpoint eval
  - checkpoint pair comparison
  - replay stability suite

### 2. Runner composition

- reuse the current eval, replay, and compare commands instead of duplicating core logic
- add a thin orchestration layer that runs the selected benchmark cases and stores artifacts under one suite root
- persist per-case and aggregate status in machine-readable form

### 3. Statistics and reporting

- compute aggregate metrics over repeated runs
- add bootstrap or similar interval estimates for core metrics such as combat win rate and reward per combat
- add concise summary payloads that are suitable for schedule promotion or leaderboard use later

### 4. CLI and report UX

- add benchmark CLI commands
- render a compact terminal summary plus a JSON summary artifact
- document the expected suite artifact tree

### 5. Tests

- test suite spec parsing
- test aggregate metric calculation and uncertainty estimates
- test report generation on fake eval outputs

## Deliverables

- benchmark suite spec and parser
- suite orchestration runner
- aggregate statistical report builder
- CLI entry points for suite execution
- automated tests around aggregation and reporting
