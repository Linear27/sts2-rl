# Issue 0.2.1-05

Status: completed on current branch.

## Title

DQN hardening

## Objective

Make the current combat DQN more stable, more sample-efficient, and easier to compare across runs.

## Problem

The current masked DQN is live-valid and checkpointed, but it is still a baseline learner. Stability and reporting are not yet strong enough for longer training schedules or reliable regression comparison.

## In Scope

- Add Double DQN
- Add n-step returns
- Add prioritized replay
- Extend training summaries with clearer learning metrics and checkpoint comparison data
- Improve evaluation reports for combat-only performance

## Out Of Scope

- Replacing DQN with PPO or APPO in this patch line
- Full-run RL redesign
- Multi-machine training orchestration

## Dependencies

- `0.2.1-01` Episode lifecycle controller
- `0.2.1-02` Replay and determinism harness
- Benefits from `0.2.1-04` but does not strictly require it

## Affected Areas

- `src/sts2_rl/train/dqn.py`
- `src/sts2_rl/train/runner.py`
- `src/sts2_rl/train/scheduler.py`
- `src/sts2_rl/cli.py`
- `tests/`

## Acceptance Criteria

- Training supports Double DQN, n-step returns, and prioritized replay through explicit configuration
- Checkpoint and summary artifacts expose the new training metrics
- Evaluation output is strong enough to compare two checkpoints without manual log digging
- Unit tests cover replay buffer behavior and target update logic

## Verification

- Automated tests for new DQN components
- Local training smoke run with checkpoint creation
- Local evaluation smoke run that compares hardened output against the old reporting path

## Delivered

- `src/sts2_rl/train/dqn.py` now supports Double DQN, n-step returns, prioritized replay, richer replay/update stats, and checkpoint schema v2 metadata.
- `src/sts2_rl/train/runner.py` now writes `learning_metrics`, `replay_metrics`, `checkpoint_comparison`, `checkpoint_metadata`, and `combat_performance` into session summaries.
- `src/sts2_rl/train/compare.py` adds repeated checkpoint-to-checkpoint comparison artifacts for combat-only evaluation.
- `src/sts2_rl/cli.py` exposes the hardened config flags and `eval combat-dqn-compare`.
- `tests/` now cover prioritized replay bias, n-step transition flushing, Double DQN target selection, and comparison/report artifacts.

## Notes

This issue comes after lifecycle and replay because those two items make model-level improvements measurable instead of anecdotal.
