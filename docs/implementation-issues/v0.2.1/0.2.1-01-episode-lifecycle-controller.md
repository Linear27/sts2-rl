# Issue 0.2.1-01

## Title

Episode lifecycle controller

## Status

Completed on the current branch.

## Objective

Turn collection, training, and evaluation from step-budget-driven loops into unattended run-aware loops.

## Problem

`v0.2` is live-validated but still behaves like a bounded stepping harness. That is enough for smoke runs, but not enough for unattended data collection or repeated training sessions because menu entry, post-run recovery, and stop conditions are still too manual.

## In Scope

- Detect and handle `MAIN_MENU`, character select, in-run states, and `GAME_OVER`
- Start a fresh run automatically when the current session expects another episode
- Return to main menu and continue after a completed or failed run
- Add explicit run-oriented limits such as `max-runs`, `max-episodes`, and `max-combats`
- Persist run counters and terminal stop reasons into `summary.json`
- Improve wait and retry behavior during screen transitions and post-combat idle periods
- Cover both collection and combat-DQN train/eval flows

## Out Of Scope

- Multi-instance orchestration changes
- New model architectures
- Determinism reporting beyond what is required for basic lifecycle correctness

## Dependencies

- None inside `v0.2.1`

## Affected Areas

- `src/sts2_rl/collect/runner.py`
- `src/sts2_rl/train/runner.py`
- `src/sts2_rl/collect/policy.py`
- `src/sts2_rl/cli.py`
- `src/sts2_rl/data/trajectory.py`
- `tests/`

## Acceptance Criteria

- Collection can run unattended for more than one complete run on the verified local single-instance configuration
- Training can recover from `GAME_OVER` to `MAIN_MENU` and continue until a run-oriented budget is exhausted
- Evaluation can execute a run-oriented budget without manual intervention
- Session summaries include run counts and a precise stop reason
- Existing step-budget flows remain available where still useful
- Unit tests cover lifecycle transitions and idle recovery behavior

## Verification

- Automated tests for menu-to-run, run-to-game-over, and game-over-to-menu transitions
- One live collection smoke run using run-oriented limits
- One live training smoke run using run-oriented limits

Current branch verification:

- `uv run pytest -q` passed with `37 passed`
- live training smoke: `stop_reason=max_combats_reached` with `completed_run_count=1` and `completed_combat_count=1`
- live collection smoke from `GAME_OVER`: recovered into a new run and stopped on `max_steps_per_instance_reached`

## Notes

This is the first blocking issue because every later `v0.2.1` item benefits from stable unattended run control.
