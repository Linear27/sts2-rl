# Issue 0.2.1-02

## Title

Replay and determinism harness

## Status

Completed on the current branch.

## Objective

Make repeatability measurable so checkpoints and policy changes can be compared under controlled conditions.

## Problem

The current stack can train and evaluate, but it does not yet provide a disciplined way to replay comparable runs or surface non-deterministic behavior. That weakens checkpoint comparison and makes regressions harder to explain.

## In Scope

- Add a fixed-configuration evaluation path for repeated runs
- Support repeated replay or re-evaluation from the same checkpoint
- Emit comparison artifacts such as action histograms, outcome summaries, and disagreement counts
- Surface repeatability or instability in a machine-readable summary
- Add CLI entry points or flags for replay-style evaluation

## Out Of Scope

- Enforcing true engine determinism
- Save-state systems
- Full simulator or headless execution

## Dependencies

- `0.2.1-01` Episode lifecycle controller

## Affected Areas

- `src/sts2_rl/train/runner.py`
- `src/sts2_rl/cli.py`
- `src/sts2_rl/data/trajectory.py`
- `artifacts/` output conventions
- `tests/`

## Acceptance Criteria

- A checkpoint can be evaluated multiple times through one explicit replay-oriented command or flow
- Output artifacts include per-run summary comparisons and action-level comparison data
- Session summaries clearly state when repeated runs diverge
- Tests cover artifact generation and summary reporting
